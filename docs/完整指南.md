## 项目概述

### 什么是 Claude 技能库？

Claude 技能库是一个智能开发引导系统，它：

- 🤖 **智能引导**：根据项目阶段自动给出建议
- 🔄 **自动化工作流**：集成开发最佳实践
- 🛠️ **工具集成**：整合 Git、测试、文档等工具
- 💬 **自然语言驱动**：用自然语言就能操作

### 适用场景

- ✅ 新项目快速启动
- ✅ 规范化开发流程
- ✅ 团队协作标准化
- ✅ 代码质量保证
- ✅ 文档自动维护

### 技术栈支持

- JavaScript/TypeScript
- Python
- Ruby/Rails
- 其他语言（通过扩展）

## 核心概念

### 1. 项目阶段

项目分为四个阶段：

```
项目初期 (Initiation)
  ↓
项目开始 (Kickoff)
  ↓
开发中 (Development)
  ↓
里程碑完成 (Milestone)
```

每个阶段有特定的任务和目标。

### 2. 智能引导系统

引导系统会：
- 追踪项目状态
- 提供下一步建议
- 展示最佳实践
- 列出可用技能

### 3. 技能命令

技能是预定义的工作流，例如：
- `/brainstorming` - 需求探索
- `/writing-plans` - 创建计划
- `/test-driven-development` - TDD 开发

### 4. 自动化 Hooks

Hooks 在特定事件触发：
- `pre_commit` - 提交前检查
- `post_commit` - 提交后更新文档

### 5. 状态追踪

系统自动记录：
- 当前项目阶段
- 最后活动
- 活动历史

## 完整工作流

### 阶段 1：项目初期

**目标**：完成环境设置

#### 步骤 1.1：克隆或创建项目

```bash
# 使用技能库
git clone https://github.com/yourusername/harness_skill.git
cd harness_skill

# 或在现有项目中使用
cd your-project
```

#### 步骤 1.2：快速初始化

```bash
bash scripts/quickstart.sh
```

这会：
1. 检查依赖
2. 创建配置文件
3. 初始化目录结构
4. 运行验证

#### 步骤 1.3：配置环境

编辑 `config/env.json`：

```json
{
  "PROJECT_NAME": "my-awesome-app",
  "PROJECT_DESCRIPTION": "一个很棒的应用",
  "GITHUB_USER": "your-username",
  "GITHUB_REPO": "your-repo",
  "GITHUB_TOKEN": "ghp_...",
  "DATABASE_URL": "mongodb://localhost:27017/mydb",
  "MODEL": "claude-sonnet-3.5",
  "DEFAULT_BRANCH": "main"
}
```

#### 步骤 1.4：验证安装

```bash
bash scripts/check_setup.sh
```

预期输出：
```
✅ Claude Code CLI 已安装
✅ Git 已安装
✅ 配置文件存在
...
✅ 所有检查通过！(16/16)
```

### 阶段 2：项目开始

**目标**：明确需求，制定计划

#### 步骤 2.1：需求探索

在 Claude 中运行：

```
/brainstorming
```

**交互流程**：

1. **探索上下文**
   - Claude 会检查现有文件
   - 了解项目背景

2. **回答问题**（一次一个）
   ```
   Q: 项目要解决什么问题？
   A: 我想构建一个任务管理应用
   
   Q: 目标用户是谁？
   A: 个人用户和小型团队
   
   Q: 有什么技术约束？
   A: 使用 React 和 Node.js
   
   Q: 成功标准是什么？
   A: 用户可以创建、分配和跟踪任务
   ```

3. **评估方案**
   
   Claude 会提出 2-3 种方案：
   
   **方案 A：快速开发**
   - 技术栈：MongoDB + Express + React
   - 优点：开发快，灵活
   - 缺点：数据一致性需要手动处理
   - 适用：MVP 快速验证
   
   **方案 B：结构化开发**
   - 技术栈：PostgreSQL + Prisma + React
   - 优点：强类型，数据一致性好
   - 缺点：需要定义 schema
   - 适用：长期维护项目
   
   **推荐**：方案 B（因为任务管理需要关系型数据）

4. **确认方案**
   
   选择方案后，Claude 会：
   - 创建设计文档 `docs/plans/YYYY-MM-DD-task-manager-design.md`
   - 记录架构决策 `docs/decisions/001-use-postgresql.md`

#### 步骤 2.2：创建实施计划

```
/writing-plans
```

生成的计划包括：

```markdown
# 任务管理器实施计划

## 任务清单
- [ ] 环境设置
  - [ ] 安装依赖
  - [ ] 配置数据库
  - [ ] 设置测试框架
- [ ] 数据库模型设计
  - [ ] User 模型
  - [ ] Task 模型
  - [ ] 关系定义
- [ ] API 开发
  - [ ] 用户认证
  - [ ] 任务 CRUD
  - [ ] 权限控制
- [ ] 前端开发
  - [ ] 登录页面
  - [ ] 任务列表
  - [ ] 任务详情
- [ ] 测试
  - [ ] 单元测试
  - [ ] 集成测试
  - [ ] E2E 测试

## 依赖关系
环境设置 → 数据库模型 → API 开发 → 前端开发 → 测试

## 关键文件
- src/models/User.js
- src/models/Task.js
- src/routes/auth.js
- src/routes/tasks.js
- src/components/TaskList.jsx
- src/components/TaskDetail.jsx

## 时间估算
- 环境设置：2小时
- 数据库模型：4小时
- API 开发：16小时
- 前端开发：20小时
- 测试：8小时
总计：50小时
```

### 阶段 3：开发中

**目标**：高效开发功能

#### 工作流 A：开发新功能

**完整流程**：

```
/brainstorming → /writing-plans → 开发 → 测试 → 审查 → 提交
```

**详细步骤**：

1. **创建功能分支**
   ```
   /using-git-worktrees
   ```
   
   这会：
   - 创建新分支 `feature/task-crud`
   - 创建独立工作目录
   - 在新窗口中打开

2. **TDD 开发**
   ```
   /test-driven-development
   ```
   
   流程：
   ```
   1. 编写失败的测试
      ↓
   2. 实现最少代码让测试通过
      ↓
   3. 重构代码
      ↓
   4. 重复
   ```
   
   示例：
   ```javascript
   // 1. 编写测试
   describe('Task API', () => {
     it('should create a new task', async () => {
       const response = await request(app)
         .post('/api/tasks')
         .send({ title: 'Test Task' });
       
       expect(response.status).toBe(201);
       expect(response.body.title).toBe('Test Task');
     });
   });
   
   // 2. 实现功能
   app.post('/api/tasks', async (req, res) => {
     const task = await Task.create(req.body);
     res.status(201).json(task);
   });
   
   // 3. 运行测试
   npm test
   
   // 4. 重构（如需要）
   ```

3. **并行开发**（可选）
   ```
   /subagent-driven-development
   ```
   
   适用于：
   - 复杂功能
   - 多个独立任务
   - 需要并行处理

4. **代码审查**
   ```
   /requesting-code-review
   ```
   
   审查内容：
   - ✅ 代码质量
   - ✅ 可读性
   - ✅ 性能
   - ✅ 安全性
   - ✅ 测试覆盖

5. **专业审查**（按语言）
   ```
   # TypeScript
   /compound-engineering:review:kieran-typescript-reviewer
   
   # Python
   /compound-engineering:review:kieran-python-reviewer
   
   # Rails
   /compound-engineering:review:kieran-rails-reviewer
   ```

6. **完成前验证**
   ```
   /verification-before-completion
   ```
   
   检查清单：
   - [ ] 所有测试通过
   - [ ] 代码审查完成
   - [ ] 文档已更新
   - [ ] 变更日志已更新
   - [ ] 无遗留 TODO
   - [ ] 符合编码规范

7. **创建 PR**
   ```
   /finishing-a-development-branch
   ```

#### 工作流 B：Bug 修复

**完整流程**：

```
/systematic-debugging → 复现 → 修复 → 验证 → 提交
```

**详细步骤**：

1. **系统化调试**
   ```
   /systematic-debugging
   ```
   
   流程：
   ```
   1. 理解问题
      - 阅读错误信息
      - 查看日志
      - 了解预期行为
   
   2. 复现 Bug
      - 确定复现步骤
      - 编写复现测试
   
   3. 定位原因
      - 使用调试器
      - 添加日志
      - 二分查找
   
   4. 提出修复方案
      - 至少 2 种方法
      - 评估优缺点
   
   5. 实施修复
      - 选择最佳方案
      - 编写代码
   
   6. 验证修复
      - 运行复现测试
      - 运行所有测试
      - 手动验证
   ```

2. **Bug 复现验证**
   ```
   /compound-engineering:workflow:bug-reproduction-validator
   ```

3. **提交修复**
   ```
   git commit -m "fix(api): 修复任务查询接口空指针异常
   
   - 添加空值检查
   - 添加错误处理
   - 添加单元测试
   
   Fixes #123"
   ```

#### 工作流 C：代码重构

1. **确保测试覆盖**
   ```bash
   npm test -- --coverage
   ```

2. **小步重构**
   - 一次只改一个地方
   - 每次改动后运行测试

3. **提交重构**
   ```bash
   git commit -m "refactor(user): 提取用户验证逻辑到独立函数"
   ```
